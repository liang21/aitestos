# 核心功能规格说明书

> 本文档基于 PRD v2.1、OpenAPI 3.0.3、UX 设计规范 v2.0、前端详细设计 v2.0 编译生成，
> 覆盖 Aitestos 智能测试管理平台的核心业务流程、页面规格与验收标准。
> 作为前端实现的唯一权威参考文档。

---

## 1. 概述与范围

### 1.1 产品愿景

AI 驱动的智能测试用例生成与管理平台。通过 AI 技术将 PRD、Figma 设计稿、API 规范等需求文档自动转化为可执行的测试用例，实现"需求即测试"的理念。

### 1.2 目标用户

| 用户群体       | 核心痛点                     | 主要场景                     |
| -------------- | ---------------------------- | ---------------------------- |
| 测试团队负责人 | 用例编写耗时长、人力成本高   | 仪表盘、AI 生成、计划管理    |
| 测试工程师     | 重复性工作多、需求变更追溯难 | 用例管理、草稿确认、测试执行 |
| 项目经理       | 测试进度不可视、覆盖率难统计 | 仪表盘、计划列表、执行报告   |

### 1.3 角色与权限矩阵

| 角色       | 标识          | 权限范围             |
| ---------- | ------------- | -------------------- |
| 超级管理员 | `super_admin` | 全部权限，含系统配置 |
| 管理员     | `admin`       | 项目内全部权限       |
| 测试工程师 | `normal`      | 用例读写、计划执行   |

| 功能模块 | 超级管理员 | 管理员 | 测试工程师 |
| -------- | :--------: | :----: | :--------: |
| 系统配置 |     ✅     |   ❌   |     ❌     |
| 项目管理 |     ✅     |   ✅   |    只读    |
| 文档上传 |     ✅     |   ✅   |     ✅     |
| 生成任务 |     ✅     |   ✅   |     ✅     |
| 草稿确认 |     ✅     |   ✅   |     ✅     |
| 用例管理 |     ✅     |   ✅   |     ✅     |
| 计划执行 |     ✅     |   ✅   |     ✅     |
| 结果录入 |     ✅     |   ✅   |     ✅     |

### 1.4 MVP 范围（v1.0）

| 模块     | 功能                | 优先级 |

| 项目管理 | 项目/模块 CRUD      | P0     |

## 2. 用户故事

### Epic 1: 项目管理

#### US-1.1 创建项目

```
作为 超级管理员
我想要 创建新项目（名称 + 前缀 + 描述）
以便于 隔离不同产品的测试资产

验收标准：
- 输入项目名称（必填，2-255 字符，唯一）
- 输入项目前缀（必填，2-4 位大写字母，唯一）
- 输入项目描述（可选）
- 创建成功后跳转到项目仪表盘
- 名称或前缀重复时 409 错误，字段级提示
```

#### US-1.2 管理模块

```
作为 管理员
我想要 在项目下创建功能模块（名称 + 缩写）
以便于 按功能组织测试用例

验收标准：
- 模块名称在项目内唯一
- 缩写 2-4 位大写字母，项目内唯一
- 删除模块时 Modal 确认，提示级联删除用例
```

### Epic 2: 知识库构建

#### US-2.1 上传 PRD 文档

```
作为 管理员
我想要 上传 PRD 文档到项目知识库
以便于 AI 基于需求生成测试用例

验收标准：
- 支持 .docx, .pdf, .md 格式（PRD 类型）
- 支持 .json, .yaml 格式（API Spec 类型）
- 上传后显示处理状态（pending → processing → completed/failed）
- 处理完成后可查看文档分块
- 处理失败显示错误原因
```

#### US-2.2 关联 Figma 设计稿

```
作为 管理员
我想要 关联 Figma 设计稿
以便于 AI 理解 UI 设计生成测试用例

验收标准：
- 输入 Figma 文件 URL
- 支持个人令牌认证方式
- 系统解析后展示节点树，支持勾选导入
- 支持选择特定页面/组件
```

### Epic 3: AI 用例生成（核心）

#### US-3.1 发起生成任务

```
作为 管理员
我想要 描述测试需求并生成用例草稿
以便于 快速获得测试用例

验收标准：
- 输入需求描述（必填，≥10 字符）
- 选择目标模块（必填）
- 可选用例数量（1-20，默认 5）
- 可选类型、优先级、场景类型
- 项目无文档时禁止生成，提示"请先上传需求文档"
- 知识库不足时弹窗警告，用户确认后可继续
- 提交后跳转任务详情页
```

#### US-3.2 查看生成结果

```
作为 测试工程师
我想要 查看生成的用例草稿
以便于 审核并决定是否采用

验收标准：
- 显示草稿列表：标题、类型 Tag、优先级 Tag、置信度 Tag
- 点击可进入草稿确认页
- 显示 AI 置信度（高/中/低）+ 引用来源
- Processing 状态显示辉光脉冲动画 + 轮询
```

#### US-3.3 确认草稿

```
作为 测试工程师
我想要 确认采用某个草稿
以便于 将其转为正式测试用例

验收标准：
- 确认前可编辑草稿所有字段
- 确认时选择目标模块
- 确认后自动生成编号（格式：前缀-缩写-日期-序号）
- 草稿状态变为 confirmed
- 支持批量确认（最多 50 条）
```

#### US-3.4 拒绝草稿

```
作为 测试工程师
我想要 拒绝不合适的草稿
以便于 反馈给 AI 改进

验收标准：
- 选择拒绝原因（重复/无关/低质量/其他）
- 可填写详细反馈（选填）
- 草稿状态变为 rejected
```

### Epic 4: 用例管理

#### US-4.1 查看用例详情

```
作为 测试工程师
我想要 查看测试用例的完整信息
以便于 了解测试内容和要求

验收标准：
- 显示所有属性（编号、标题、前置条件、步骤、预期结果、类型、优先级、状态）
- AI 生成的用例显示 AI 元数据区（生成任务、置信度、引用来源、模型版本）
- 源文档已变更时显示 ⚠️ 警告
- 显示执行历史
```

#### US-4.2 编辑用例

```
作为 测试工程师
我想要 修改测试用例
以便于 更新测试内容

验收标准：
- 可修改所有属性
- 修改后记录更新时间
- AI 生成的用例可手动调整
```

#### US-4.3 需求追溯

```
作为 测试工程师
我想要 查看用例关联的需求来源
以便于 理解测试依据

验收标准：
- AI 生成的用例显示引用的文档块
- 显示相似度得分，可点击查看原文
- 源文档变更或删除时有明确提示
```

### Epic 5: 测试执行

#### US-5.1 创建测试计划

```
作为 管理员
我想要 创建测试计划并选择用例
以便于 组织本轮测试

验收标准：
- 输入计划名称（必填，2-255 字符）
- 从用例库选择用例（支持按模块/类型/优先级筛选）
- 创建后状态为 draft
```

#### US-5.2 执行测试并录入结果

```
作为 测试工程师
我想要 执行测试并记录结果
以便于 跟踪测试进度

验收标准：
- 快捷录入：内联 Select 选择状态（pass/fail/block/skip）+ Toast 确认 + 3s 撤销
- 详细录入：Modal 填写状态 + 备注
- 支持批量录入
- 录入后更新计划统计
```

#### US-5.3 查看执行历史

```
作为 测试工程师
我想要 查看用例的历史执行记录
以便于 分析问题趋势

验收标准：
- 用例详情页显示执行历史表格
- 计划名称可点击跳转
- 显示执行结果、执行人、时间、备注
```

### 跨模块用户故事

#### US-C01 JWT Token 无感刷新

```
作为 已登录用户
我想要 在 Token 过期时系统自动刷新而无需重新登录
以便于 长时间操作期间不会丢失工作状态

验收标准：
- 前端检测到 401 时自动使用 refresh_token 刷新
- 刷新成功后自动重放原始请求
- 并发请求时仅触发一次刷新，其余排队等待
- 刷新失败时跳转登录页，保留来源路由
```

## 3. 核心业务流程

### 3.1 Flow 1: AI 生成用例全流程

```
知识库为空？
  ├── 是 → [知识库页] 上传 PRD 文档 → 等待解析(completed) ──┐
  └── 否 ─────────────────────────────────────────────────────┘
                                                              ↓
[AI 生成页] 新建生成任务 → 填写需求描述 + 选模块 → 提交
        ↓ POST /generation/tasks
[任务详情页] 等待处理(processing) → 轮询每 5s
        ↓ GET /generation/tasks/{id}
    ┌─── completed ───┐
    │                  │
    ▼                  ▼
[草稿列表]        [任务失败] → 重试 → 回到等待
  │ 查看每条草稿
  ▼
[草稿确认页] 编辑内容 → 查看引用来源
  ├── 确认 → POST /generation/drafts/{id}/confirm → [用例详情页]
  ├── 拒绝 → POST /generation/drafts/{id}/reject → 回到草稿列表
  └── 保存 → 暂存修改

涉及页面：知识库文档列表、文档上传 Modal、AI 生成任务列表、新建生成任务、任务详情、草稿确认
涉及 API：POST /knowledge/documents → GET /knowledge/documents/{id} → POST /generation/tasks
         → GET /generation/tasks/{id}(轮询) → GET /generation/tasks/{id}/drafts
         → POST /generation/drafts/{id}/confirm
```

### 3.2 Flow 2: 测试执行全流程

```
[计划列表页] 新建计划
        ↓
[新建计划页] 填写名称 + 选择用例 → 提交
        ↓ POST /plans
[计划详情页] 状态: draft
  │ 点击"开始执行"
  ↓ PATCH /plans/{id}/status { status: active }
[计划详情页] 状态: active
  │ 逐条录入执行结果
  │   ├── 快捷录入: 内联 Select → Toast 确认
  │   └── 详细录入: 点击"录入" → Modal → 提交
  │ POST /plans/{id}/results
  │
  │ 全部执行完毕 → 点击"标记完成"
  ↓ PATCH /plans/{id}/status { status: completed }
[计划详情页] 状态: completed → 查看统计（通过率、失败数）
  │ 可选: 点击"归档"
  ↓ PATCH /plans/{id}/status { status: archived }

涉及页面：计划列表、新建计划、计划详情
涉及 API：POST /plans → GET /plans/{id} → PATCH /plans/{id}/status(3次)
         → POST /plans/{id}/results(多次)
```

### 3.3 Flow 3: 需求变更响应

```
[知识库页] 发现文档需更新
  │ 点击"重新解析"（或重新上传）
  ↓ POST /knowledge/documents（覆盖）
[文档详情页] 等待重新解析 → completed
  │ 文档块 ID 可能变化
  ↓
[用例详情页] 查看已关联用例
  │ AI 元数据区显示 ⚠️ "源文档已变更"
  │ 用户手动核验用例内容
  │   ├── 内容仍有效 → 忽略警告
  │   └── 内容需更新 → 编辑用例
  ↓ PUT /testcases/{id}

涉及页面：知识库列表、文档详情、用例详情
涉及 API：POST /knowledge/documents → GET /knowledge/documents/{id}/chunks
         → GET /testcases/{id} → PUT /testcases/{id}
```

### 3.4 用例编号规则

**格式**：`{项目前缀}-{模块缩写}-{YYYYMMDD}-{001-999}`

**示例**：`ECO-USR-20260402-001`

| 组成部分 | 来源               | 说明                       |
| -------- | ------------------ | -------------------------- |
| 项目前缀 | 创建项目时用户输入 | 2-4 位大写字母，全局唯一   |
| 模块缩写 | 创建模块时用户输入 | 2-4 位大写字母，项目内唯一 |
| 日期     | 系统自动生成       | YYYYMMDD                   |
| 序号     | 系统自动递增       | 当日该模块下 001-999       |

### 3.5 RAG 检索与降级策略

**检索流程**：需求描述 → Embedding 向量化 → Milvus Top-K 检索 → 组装 Prompt → LLM 生成

| 场景                  | 处理方式                                                                     |
| --------------------- | ---------------------------------------------------------------------------- |
| 检索结果 < 1 个文档块 | 弹窗提示"知识库内容不足，生成质量可能较低"，用户确认后继续，置信度强制为"低" |
| 完全无知识库          | 禁止发起生成任务，提示"请先上传需求文档"                                     |

### 3.6 AI 置信度定义

| 级别 | 判定条件                                    | UI 展示     |
| ---- | ------------------------------------------- | ----------- |
| 高   | 检索到 ≥2 个相关文档块，且语义相似度 > 0.8  | 🟢 绿色标签 |
| 中   | 检索到 1 个相关文档块，且语义相似度 0.5-0.8 | 🟡 黄色标签 |
| 低   | 无匹配文档块，纯 LLM 通用知识生成           | 🔴 红色标签 |

### 3.7 文档生命周期策略

**文档更新（重新解析）**：

- 保留用例引用记录，标记为"源文档已变更"
- 用例详情页显示 ⚠️ 警告
- 不自动删除或修改已确认的用例

**文档删除**：

- 采用软删除，保留引用关系
- 用例详情页显示"源文档已删除"
- 提供"移除引用"操作供用户手动清理

## 4. 页面规格

> 每个页面规格包含：路由、布局、组件、API、交互行为。

### 4.1 认证模块

#### 4.1.1 登录页 (/login)

**布局**：左右分栏（55% : 45%），AuthLayout。

**左侧品牌区**：Mesh Gradient 背景（#7B61FF → #5A3DC0 → #3B1FA0），浮动几何图形，产品 Slogan。

**右侧表单区**：

| 字段 | 组件                | 必填 | 验证       |
| ---- | ------------------- | ---- | ---------- |
| 邮箱 | Arco Input          | 是   | email 格式 |
| 密码 | Arco Input.Password | 是   | 非空       |

**交互流程**：

1. 填写邮箱和密码 → 点击"登录"
2. 按钮 loading 态，禁用重复点击
3. 成功 → token 写入 localStorage → 跳转 `/projects`
4. 失败 → Arco Alert (error) "邮箱或密码错误"

**API**：`POST /auth/login` → `{ access_token, refresh_token, user }`

#### 4.1.2 注册页 (/register)

**布局**：同登录页左右分栏，AuthLayout。

| 字段     | 组件                | 必填 | 验证           |
| -------- | ------------------- | ---- | -------------- |
| 用户名   | Arco Input          | 是   | 3-32 字符      |
| 邮箱     | Arco Input          | 是   | email 格式     |
| 密码     | Arco Input.Password | 是   | ≥8 字符        |
| 确认密码 | Arco Input.Password | 是   | 与密码一致     |
| 角色     | Arco Select         | 是   | admin / normal |

**注意**：注册页不提供"超级管理员"选项。

**错误处理**：409 → 字段级错误"邮箱已存在"或"用户名已存在"。成功 → Message.success → 跳转登录页。

**API**：`POST /auth/register` → `UserJSON`

### 4.2 项目管理模块

#### 4.2.1 项目列表页 (/projects)

**布局**：AppLayout，搜索 + Card 网格（3 列）。

**卡片**：顶部 3px 颜色装饰条（基于前缀 hash）+ 项目名称(h3) + 前缀 Tag(monospace) + 描述(2行截断) + 统计行 + 操作按钮。悬浮时 shadow-md + 底部按钮 fadeIn。

#### 4.2.2 创建项目模态框

**触发**：点击"新建项目" → Modal (600px)。

| 字段     | 组件          | 必填 | 验证                            |
| -------- | ------------- | ---- | ------------------------------- |
| 项目名称 | Arco Input    | 是   | 2-255 字符，唯一                |
| 项目前缀 | Arco Input    | 是   | 2-4 位大写字母 `^[A-Z]+$`，唯一 |
| 项目描述 | Arco TextArea | 否   | —                               |

**交互**：前缀输入时实时校验格式，失焦校验唯一性。409 → 字段错误。成功 → Message.success + 关闭 + 刷新。

**API**：`POST /projects` → `Project`

#### 4.2.3 项目仪表盘 (/projects/:id)

## 6. 技术实现约束

> 本节仅列出约束原则和引用，具体代码详见 `specs/frontend-detailed-design.md`。

### 6.1 架构与目录结构

→ 参考 `frontend-detailed-design.md` 第一章

- Feature-Based 架构，每个模块内聚 components/hooks/services
- 依赖规则：Page → Feature hooks → Feature services → `@/lib/request`
- `components/` 禁止引用 `features/`
- `services/` 禁止引用 `store/` 或 `hooks/`
- Feature A 禁止引用 Feature B

### 6.2 路由设计

→ 参考 `frontend-detailed-design.md` 第二章（含完整路由表和 lazy loading 配置）

- 完整路由表见该文档 §2.1
- RouteGuard 组件处理认证和权限
- JWT `exp` 字段判断过期
- admin+ 路由使用 `requireAdmin` 属性

### 6.3 状态管理

→ 参考 `frontend-detailed-design.md` 第三章

- **Zustand**：仅管理 UI 状态（`useAppStore`）和认证 token（`useAuthStore`）
- **React Query**：所有服务端数据
- 禁止将服务端数据存入 Zustand
- Query Key 工厂模式确保缓存粒度
- 轮询使用 `refetchInterval`（如 AI 任务状态每 5s）

### 6.4 API 层设计

→ 参考 `frontend-detailed-design.md` 第四章

- Axios 实例封装类型安全的 wrapper（`get<T>`, `post<T,R>` 等）
- Token 刷新机制：401 → 自动刷新 → 重放请求 → 并发排队
- 每个 Feature 独立 service 文件
- 禁止组件直接 import axios

### 6.5 类型定义

→ 参考 `frontend-detailed-design.md` §4.2（enums.ts + api.ts）

- 所有枚举类型定义在 `src/types/enums.ts`
- 所有 API 请求/响应类型在 `src/types/api.ts`
- 禁止使用 `any` 类型

### 6.6 错误处理策略

→ 参考 `frontend-detailed-design.md` §4.1 + PRD §4.1.1

| HTTP 状态码 | 处理方式     | UI 反馈                       |
| ----------- | ------------ | ----------------------------- |
| 400         | 表单验证提示 | 字段下方红色错误              |
| 401         | 跳转登录页   | 清除 token + redirect         |
| 403         | 无权限提示   | Notification.warning          |
| 404         | 404 页面     | "返回首页"按钮                |
| 409         | 冲突提示     | 字段级错误                    |
| 500         | 服务器错误   | Notification.error + 重试按钮 |

**LLM 超时**：单次 30s，总任务 120s。失败 → 任务状态 failed + 手动重试。

### 6.7 设计令牌速查

→ 参考 `ux-design-spec.md` 第二部分

**品牌色**：Primary `#7B61FF` | AI Glow `rgba(123,97,255,0.15)` | AI Gradient `linear-gradient(135deg, #7B61FF, #9B7BFF)`

**字体**：标题 `DM Sans` | 中文 `system-ui` | 编号 `JetBrains Mono`

**间距基础**：4px，常用 8/12/16/20/24px
