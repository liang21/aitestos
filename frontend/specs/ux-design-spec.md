# 第一部分：设计总纲

## 1.1 设计目标

1. **企业级专业感**：界面布局紧凑有序，信息层级清晰，操作路径短，符合企业级测试管理工具的专业定位
2. **效率优先**：高频操作一步到位，批量操作流畅，键盘友好，减少不必要的页面跳转
3. **AI 智能感**：AI 生成相关界面使用品牌紫色高亮，置信度可视化，引用来源透明可追溯

## 1.2 设计原则

| 原则             | 说明                                                                                           |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| **一致性**       | 相同功能使用相同的交互模式、视觉样式和文案。所有列表页共享统一的表格规范，所有表单共享验证规范 |
| **可预测性**     | 操作结果可预期。按钮文案清晰表达行为，危险操作需二次确认，不可逆操作标注后果                   |
| **效率优先**     | 高频路径最短。支持批量操作、快捷键、双击进入详情、内联编辑等提效手段                           |
| **渐进式复杂度** | 基础功能开箱即用，高级功能折叠隐藏（如 AI 高级选项），避免界面复杂度过高                       |
| **数据驱动**     | 关键指标前置展示（仪表盘），状态色彩编码直观，趋势图表辅助决策                                 |
| **可访问性**     | 满足 WCAG 2.1 AA 标准，色彩对比度达标，支持键盘导航，语义化 HTML                               |

## 1.3 目标用户画像

| 用户群体       | 核心痛点                     | 主要使用场景                 | 关键诉求             |
| -------------- | ---------------------------- | ---------------------------- | -------------------- |
| 测试团队负责人 | 用例编写耗时长、人力成本高   | 仪表盘、AI 生成、计划管理    | 提升效率、可视化进度 |
| 测试工程师     | 重复性工作多、需求变更追溯难 | 用例管理、草稿确认、测试执行 | 自动化生成、快速追溯 |
| 项目经理       | 测试进度不可视、覆盖率难统计 | 仪表盘、计划列表、执行报告   | 实时数据、质量指标   |

| 技术              | 版本 | 用途       |
| ----------------- | ---- | ---------- |
| React             | 19.2 | UI 框架    |
| Arco Design React | 2.66 | 组件库     |
| Tailwind CSS      | 4.x  | 工具类样式 |
| Zustand           | 5.x  | 状态管理   |
| React Router      | 7.x  | 路由       |

### 组件策略

| 类别                  | 策略                                 | 示例                                                                                                                                                                                                                                                            |
| --------------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Arco 原生直接使用** | 直接引用，遵循 Arco 默认交互         | Button、Input、Select、Table、Form、Modal、Drawer、Tag、Breadcrumb、Menu、Layout、Tree、Upload、Spin、Skeleton、Progress、Steps、Statistic、Pagination、Radio、Checkbox、Collapse、Popover、Popconfirm、Message、Notification、Avatar、Dropdown、Badge、Tooltip |
| **封装组件**          | 基于 Arco 组件二次封装，统一业务交互 | ArrayEditor（数组编辑器）、SearchTable（搜索表格）、StatusTag（状态标签）、StatsCard（统计卡片）、SplitPanel（分栏面板）、JsonEditor（JSON 编辑器）、ReferencePanel（引用来源面板）                                                                             |
| **自定义组件**        | Arco 无对应组件，完全自定义          | 用例选择面板、节点选择树（Figma）、草稿确认编辑器                                                                                                                                                                                                               |

### Arco ConfigProvider 与 Tailwind 职责边界

- **Arco ConfigProvider**：负责品牌色、圆角、组件级主题定制（通过 `theme` 属性修改 CSS 变量）
- **Tailwind CSS**：负责布局间距、响应式断点、自定义工具类（通过 `@theme` 指令定义 design tokens）
- **不重叠原则**：组件内部样式用 Arco 主题系统，页面布局和间距用 Tailwind

## 4.3 核心用户任务流程（User Task Flow）

> 以下流程描述用户从起点到终点的完整操作路径，标注涉及的页面、API 和状态流转。

### 流程 1：AI 生成用例全流程

```
知识库为空？
  ├── 是 → [知识库页] 上传 PRD 文档 → 等待解析(completed) ──┐
  └── 否 ─────────────────────────────────────────────────────┘
                                                              ↓
[AI 生成页] 新建生成任务 → 填写需求描述 + 选模块 → 提交
        ↓ POST /generation/tasks
[任务详情页] 等待处理(processing) → 轮询每 5s
        ↓ GET /generation/tasks/{id}
    ┌─── completed ───┐
    │                  │
    ▼                  ▼
[草稿列表]        [任务失败] → 重试 → 回到等待
  │ 查看每条草稿
  ▼
[草稿确认页] 编辑内容 → 查看引用来源
  ├── 确认 → POST /generation/drafts/{id}/confirm → [用例详情页]
  ├── 拒绝 → POST /generation/drafts/{id}/reject → 回到草稿列表
  └── 保存 → 暂存修改
```

**涉及页面**：知识库文档列表、文档上传 Modal、AI 生成任务列表、新建生成任务、任务详情、草稿确认
**涉及 API**：POST /knowledge/documents → GET /knowledge/documents/{id} → POST /generation/tasks → GET /generation/tasks/{id}（轮询）→ GET /generation/tasks/{id}/drafts → POST /generation/drafts/{id}/confirm

### 流程 2：测试执行全流程

```
[计划列表页] 新建计划
        ↓
[新建计划页] 填写名称 + 选择用例 → 提交
        ↓ POST /plans
[计划详情页] 状态: draft
  │ 点击"开始执行"
  ▼ POST /plans/{id}/status { status: active }
[计划详情页] 状态: active
  │ 逐条录入执行结果
  │   ├── 快捷录入: 内联 Select → Toast 确认
  │   └── 详细录入: 点击"录入" → Modal → 提交
  │ POST /plans/{id}/results
  │
  │ 全部执行完毕 → 点击"标记完成"
  ▼ POST /plans/{id}/status { status: completed }
[计划详情页] 状态: completed → 查看统计（通过率、失败数）
  │ 可选: 点击"归档"
  ▼ POST /plans/{id}/status { status: archived }
```

**涉及页面**：计划列表、新建计划、计划详情
**涉及 API**：POST /plans → GET /plans/{id} → PATCH /plans/{id}/status（3 次）→ POST /plans/{id}/results（多次）

### 流程 3：需求变更响应流程

```
[知识库页] 发现文档需更新
  │ 点击"重新解析"（或重新上传）
  ▼ POST /knowledge/documents（覆盖）
[文档详情页] 等待重新解析 → completed
  │ 文档块 ID 可能变化
  ▼
[用例详情页] 查看已关联用例
  │ AI 元数据区显示 ⚠️ "源文档已变更"
  │ 用户手动核验用例内容
  │   ├── 内容仍有效 → 忽略警告
  │   └── 内容需更新 → 编辑用例
  ▼ PUT /testcases/{id}
```

**涉及页面**：知识库列表、文档详情、用例详情
**涉及 API**：POST /knowledge/documents → GET /knowledge/documents/{id}/chunks → GET /testcases/{id} → PUT /testcases/{id}

## 6.3 全局状态与路由守卫

### Zustand Store 结构

```typescript
interface AppStore {
  // 用户
  user: { id: string; username: string; email: string; role: string } | null
  // 当前项目
  currentProject: { id: string; name: string; prefix: string } | null
  // UI 状态
  sidebarCollapsed: boolean
  // 通知
  pendingDraftCount: number
  notificationCount: number
}
```

### Token 管理

- 存储：localStorage `token` + `refresh_token`
- 拦截器：Axios request interceptor 自动附加 `Authorization: Bearer {token}`
- 刷新：401 响应时调用 `POST /auth/refresh`，成功则重试原请求，失败则清除 token 并跳转 `/login`

### 路由守卫

| 条件                    | 行为                          |
| ----------------------- | ----------------------------- |
| 未登录 + 访问需认证页面 | 重定向到 `/login`             |
| 已登录 + 访问 `/login`  | 重定向到 `/projects`          |
| 401 API 响应            | 清除 token，重定向到 `/login` |

### 通知系统设计

顶栏通知铃铛点击后弹出 `Arco Popover` 通知面板：

```
┌──────────────────────────────────┐
│  通知                    全部已读  │
├──────────────────────────────────┤
│  ✨ AI 用例生成完成               │
│  产出 5 条草稿         2 分钟前    │
│  ─────────────────────────────── │
│  📄 文档解析完成: 登录PRD         │
│  解析出 12 个分块      10 分钟前   │
│  ─────────────────────────────── │
│  ❌ 文档解析失败: API接口文档      │
│  格式不支持            1 小时前    │
├──────────────────────────────────┤
│               [查看全部通知]       │
└──────────────────────────────────┘
```

**通知数据模型**：

```typescript
interface Notification {
  id: string
  type: 'ai_complete' | 'doc_complete' | 'doc_failed' | 'case_updated'
  title: string // 通知标题
  content: string // 通知内容摘要
  read: boolean // 是否已读

  createdAt: string // 时间戳
}
```

**交互规则**：

- 未读通知：左侧显示蓝色圆点标记，背景微灰
- 点击通知项：标记已读 + 跳转 `actionUrl`

- 最多显示最近 10 条，超出部分点击"查看全部通知"

| 字段交互   | 每个字段前增加 Checkbox：勾选表示"修改此字段"，未勾选表示"保持不变" |
| 确认文案   | "将修改 N 条记录的{已勾选字段描述}"                                 |
| 操作按钮   | 取消 + "确认修改"（primary）                                        |

## 12.7 级联删除保护

对于会影响关联数据的删除操作，使用 Modal（非 Popconfirm）进行更严格的二次确认：

| 操作             | 确认方式     | 提示内容                                                                                                                        |
| ---------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------- |
| 删除用例（单条） | `Popconfirm` | "确认删除该用例？"                                                                                                              |
| 删除用例（批量） | `Modal`      | "确认删除选中的 N 条用例？此操作不可恢复"                                                                                       |
| 删除模块         | `Modal`      | "删除模块'{名称}'将同时删除其下 N 条测试用例和 M 条执行记录。此操作不可恢复。" + 输入框"请输入模块名称'{名称}'确认删除"         |
| 删除项目         | `Modal`      | "删除项目'{名称}'将同时删除其下所有模块、用例、计划、文档和执行记录。此操作不可恢复。" + 输入框"请输入项目名称'{名称}'确认删除" |

> 输入框确认机制类似 GitHub 删除仓库：用户需手动输入资源名称，输入完全匹配时"确认删除"按钮才可点击。

## 13.1 断点定义

| 名称    | 宽度范围        | 主要适配                     |
| ------- | --------------- | ---------------------------- |

| Laptop  | 1024px - 1279px | 侧边栏默认折叠               |
| Tablet  | 768px - 1023px  | 简化布局，侧边栏覆盖模式     |
| Mobile  | < 768px         | 不主动支持，保证不崩溃       |

## 13.2 各断点适配策略

### 侧边栏

| 断点               | 行为                                 |
| ------------------ | ------------------------------------ |
| Desktop (≥1280)    | 默认展开 220px                       |
| Laptop (1024-1279) | 默认折叠 64px，可手动展开            |
| Tablet (768-1023)  | 隐藏，hamburger 按钮呼出 Drawer 覆盖 |
| Mobile (<768)      | 同 Tablet                            |

### 分栏面板（ResizeBox.Split）

| 断点    | 行为                               |
| ------- | ---------------------------------- |
| Desktop | 正常左右分栏                       |
| Tablet  | 左侧面板变为顶部可折叠区域         |
| Mobile  | 全部堆叠，树形筛选变为 Select 下拉 |

### 表格

| 断点    | 行为                           |
| ------- | ------------------------------ |
| Desktop | 显示所有列                     |
| Laptop  | 隐藏次要列（创建人、更新时间） |
| Tablet  | 切换为卡片列表视图             |
| Mobile  | 简化卡片列表                   |

### 模态框

| 断点    | 行为                           |
| ------- | ------------------------------ |
| Desktop | 居中弹窗                       |
| Tablet  | 底部 Drawer 模式（从底部弹出） |
| Mobile  | 全屏页面                       |

### 表单

| 断点             | 行为     |
| ---------------- | -------- |
| Desktop / Laptop | 双列布局 |
| Tablet / Mobile  | 单列布局 |

## 13.3 关键声明

- **本产品以 Desktop（≥1280px）为主要设计目标**
- Mobile 端不保证完整功能体验，但需确保不出现布局崩溃
- 核心操作流程（用例管理、AI 生成、测试执行）在 Desktop 端完整可用
- 响应式适配优先级：Desktop > Laptop > Tablet > Mobile

## 14.3 关键设计决策记录

| 编号 | 决策                                         | 理由                                                | 影响范围           |
